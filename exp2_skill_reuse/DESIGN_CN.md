# 实验二：技能复用与生成 token 效率（假设二）

## 假设

**H2**：成功轨迹可抽象为可复用技能；在**同一类任务**上，**复用技能模板**比**从零生成**更省生成成本。

操作化预测：**completion_tokens（模型输出 token）** 在「技能条件」下显著低于「从零条件」。

## 自变量

| 水平 | 含义 |
|------|------|
| **从零（scratch）** | 仅给任务描述与与实验一一致的 `move`/`range` 协议，不提供技能段落。 |
| **技能（skill）** | 在相同任务描述前，增加一段**固定、版本化**的「曼哈顿无障碍网格」技能说明 + **已填好方向**的两行骨架，模型主要只需填两个 `range` 整数。 |

两类任务在**地图族**上一致：均为 **5×5 全空地**（`grid` 全 0），仅 **起点/终点** 变化，保证都存在**无墙曼哈顿路径**，且技能在逻辑上**适用**。

## 因变量（主 / 次）

1. **主指标**：`completion_tokens`（API 返回 `usage.completion_tokens`）。对应「生成答案的代价」，与假设表述「生成 token 数」一致。
2. **次指标**：`prompt_tokens`、`total_tokens`（技能条件通常 **prompt 更长**，总 token 是否仍下降需在结果里一并报告）。

若某次响应缺少 `usage` 字段，记为 `null` 并在汇总时剔除或单列。

## 控制

- 同一模型、同一 `temperature`（建议 0）、同一 `tasks.json`。
- **顺序效应**：每个 `task_id` 内可对 scratch/skill **随机先后**（脚本用固定 seed 可复现），或**整块交替**；分析时以任务为单元配对比较（推荐 **配对**：同一题两种条件各测一次）。

## 样本量与统计

- 建议 **每条件每题各 1 次** → 每题 2 条请求；题数 **20+**（本目录 `tasks.json` 默认 20），可再扩写 JSON。
- 报告：各条件 `completion_tokens` 的均值、标准差、**配对差值**（scratch − skill）的均值与 95% 自助置信区间或配对 t 检验（若差值近似正态）。
- 可附：**到达终点正确率**（复用 `exp1_code_verify` 的 `grade_mode_b`），说明技能条件是否在「更短」同时仍满足任务约束。

## 与实验一的关系

- 实验一：过程验证（文本 vs 可执行）对**正确率**。
- 实验二：**在代码范式内**，比较**信息结构**（是否给技能模板）对**生成长度**；技能文本计入 prompt，结论聚焦 **completion** 或明确写清 **total** 口径。

## 伦理与可复现

- `SKILL_VERSION` 写入每条 `jsonl` 结果，技能段落变更时 bump 版本。
- 不在仓库提交 API Key；`config.json` 见 `.gitignore`。
