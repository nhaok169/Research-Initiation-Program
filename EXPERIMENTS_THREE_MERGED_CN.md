## 实验一（H1）：网格寻径，文本动作 vs `move` 代码

### 设计思路（详细）

- **任务**：若干带墙网格实例；给定 ASCII 图 + `grid` JSON + 起点终点，要求模型输出**到达终点**的方案。
- **方式 A**：只允许自然语言描述移动（步数、方向词等），评测时用**启发式解析**成 `up/down/left/right` 序列，在仿真环境 `GridEnv` 里逐步执行，看终止位置是否等于终点。
- **方式 B**：只允许在 ```python 代码块中写**受限代码**：顶层仅有 `move("方向")` 与 `for _ in range(常数): ...`，由 **AST 白名单**校验后，在注入 `move` 的沙箱里执行，再在**同一环境**里判是否到终点。
- **成功判据**：**终止状态到终点**（不是「代码能跑」即满分），避免把路径错误但无异常的输出判对。

### 为什么这样设计

1. **对齐研究问题**：要比较的是「规划是否可靠落地」，因此 B 必须把状态更新交给**与环境一致的确定性** `move`，而不是让模型口头描述一套、实际另一套。
2. **沙箱偏严**：禁止 `def/import/赋值/...`，防止模型自定义 `move` 或绕过环境，保证评测与「真实可部署协议」同构。
3. **A 侧用解析+仿真**：自然语言无法唯一形式化，用规则抽取动作序列是**可复现的折中**；局限是未覆盖的表述会判错，重要结论可配合人工抽检。

### 跑出的结果（当前最佳：overnight 完整跑批）

- **设置**：固定任务集 **N=100**，完成整批 API 跑批后由 `grade_jsonl.py` 汇总（结果文件 `exp1_code_verify/graded_overnight.jsonl`）。
- **汇总**：

| 模式 | 到达终点正确率 |
|------|----------------|
| A（文本动作） | **0.07** |
| B（受限代码） | **0.06** |

- **解读**：在更大样本下，A/B 都偏低且接近，说明仅靠当前模型与提示，导航正确率仍受限；H1 更适合在更强模型、分难度任务上继续验证。

## 实验二（H2）：同一网格类任务，从零写代码 vs 技能模板

### 设计思路（详细）

- **任务域**：**5×5 全空地**，仅**起点/终点**变化，保证存在简单曼哈顿路径，避免地图拓扑干扰。
- **从零（scratch）**：在地图描述外**不提供**结构化解法模板，要求只输出符合实验一 B 协议的 ```python。
- **技能（skill）**：提供**版本化**技能段（如两行 `for i in range(____): move("…")` 的骨架），方向按起终点预先定好，模型主要**填两个非负整数**。
- **流程**：每个任务各调用 scratch 与 skill 各一次；任务内先后**随机**（种子可配），输出写入 `exp2_results.jsonl`，记录 `completion_tokens` 等；可用实验一的 `grade_mode_b` 得到 **`reach_goal`**，避免「更短但全错」。

### 为什么这样设计

1. **隔离变量**：比较的是「信息是否以**可复用技能**形式给出」，而不是「会不会走迷宫」；全空地使**技能可声明恒成立**（曼哈顿分解）。
2. **主指标用 completion_tokens**：直接对应「生成成本」假设；同时报告 **reach_goal**，把**效率与质量**绑在一起讨论。
3. **技能故意具体**：当前 v1 几乎把解法空间收到两行，**completion 会极低**在预期内；论文/计划中应写明这是**强技能**下的上界现象，泛化到技能库检索需后续版本。

### 跑出的结果（当前最佳：overnight 完整跑批）

- **设置**：**N=100** 对任务，完整跑批后 `summarize_exp2.py` 输出（结果文件 `exp2_skill_reuse/exp2_results_overnight.jsonl`）。

| 指标 | scratch | skill |
|------|---------|--------|
| completion_tokens 均值 | 4055.84（标准差大） | 26.04（几乎常数） |
| 配对平均 scratch − skill（completion） | **4029.8**（skill 远短） | — |
| reach_goal 率 | **0.02** | **0.28** |

- **解读**：在 100 对样本上，技能侧仍显著降低生成长度（约降到 scratch 的 0.64%），并保持更高到达率，较强支持 H2。

## 实验三（H3）：图表/表格阅读，文本答案 vs 受限 `ans=` 代码

### 设计思路（详细）

- **要验证的迁移命题**：在实验一里行之有效的思路——**把关键推理交给窄语言面的可执行片段、在沙箱里求值再对照 gold**——能否用到**读柱状图 / 读表**类任务。
- **刺激呈现**：**不依赖像素图**也可跑通——用 **ASCII 柱图 + `series` JSON**，和/或 **ASCII 表 + `table` JSON**；题干为中文问答（如 argmax 索引、行和、列和最大索引等）。表格题**不画柱图**，并说明 `series` 为空，避免与「导航」叙事混淆。
- **方式 A**：只要求自然语言给出**最终数值**；评测取回复中**最后一个可解析数字**与 gold 比（容差规则见实现）。
- **方式 B**：```python 中**恰好一条** `ans = <expr>`；`expr` 经 AST 白名单，仅允许对注入的 `series`/`table` 做安全运算及少量函数（如 `argmax(series)`、`row_sum(table, k)` 等），执行后取 `ans` 与 gold 比。
- **分析**：除总体 A/B 正确率外，可按 `task_type` 分层（`summarize_exp3.py`），看**纯读数**与**聚合/索引**子集上差异。

### 为什么这样设计

1. **与 H1 同构**：自变量仍是 **「纯语言 end-to-end」vs「窄代码 + 执行」**，便于写进同一条叙事：**过程可验证**从「移动」推广到「算术/索引」。
2. **沙箱哲学一致**：与实验一「只 move + range」一样，实验三「只 `ans=` + 白名单表达式」，**缩小投机面**，结论可归因到协议而非任意 Python。
3. **ASCII/JSON 是刻意选择**：先在与游戏任务相同的**纯文本 API** 上完成协议与评测闭环；若将来接 VLM，同一 **gold 与题型**可迁移到渲染图，作为扩展而非本稿必选。

### 跑出的结果（当前最佳：overnight 完整跑批）

- **设置**：`N=180`，结果文件 `exp3_chart_transfer/exp3_graded_overnight.jsonl`。
- **总体准确率**：
  - `mode_a_accuracy` = **0.6667**
  - `mode_b_accuracy` = **0.6389**
- **分题型现象（见 `summarize_exp3.py` 输出）**：
  - B 在 `argmax_*`、`max_minus_min`、`sum_all` 等结构化算子题上明显更稳；
  - A 在 `row_sum`、`col_sum`、`mid_ratio_percent` 上反而更高，提示当前 B 提示词/沙箱算子定义仍需微调。

2. **模型与日期**：换模型、换温度后数字会变，材料中应**注明模型名与跑批日期**。
3. **实验一的 A 侧**：自动解析非完备，讨论中保留**抽检与 reason 日志**空间。
4. **实验二的技能强度**：当前技能极强，结论宜表述为「在该类强模板下」而非任意技能库。
